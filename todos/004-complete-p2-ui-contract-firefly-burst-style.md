## Problem Statement
Firefly burst state now accepts both `success` and `error` event types, but stylesheet only defines animation for `firefly-burst-success`. Error bursts never resolve to a concrete visual style and silently degrade user feedback.

## Findings
- `useFirefly` exposes burst state type `success | error` and calls `triggerBurst` with `error` in success/error paths.
- `FireflySignal` maps `burstType` to CSS class `firefly-burst-${burstType}`.
- `src/styles/firefly.css` currently includes `@keyframes firefly-burst-success` and `.firefly-burst-success` only.
- Passing `error` produces an undeclared class with no animation, making error outcomes visually indistinguishable from idle.

### Evidence
- `src/hooks/useFirefly.ts:153-163`
- `src/ui/views/BranchView.tsx:685-690` (error/success triggering)
- `src/ui/components/FireflySignal.tsx:74-79`
- `src/styles/firefly.css` around burst keyframe definitions

## Proposed Solutions
### Option 1: Add `firefly-burst-error` styling (Recommended)
- **Effort:** Small
- **Risk:** Low
- **Pros:** Restores intended visual semantics quickly.
- **Approach:**
  - Add error keyframes/color token path and `.firefly-burst-error` class.

### Option 2: Restrict burst domain to implemented styles
- **Effort:** Small
- **Risk:** Low
- **Pros:** Prevents invalid states.
- **Approach:**
  - Narrow burst union to `success` until error style is designed.
  - Map unknown values to no-op or fallback.

### Option 3: Centralized semantic feedback map
- **Effort:** Medium
- **Risk:** Low
- **Pros:** Guards against future drift between types and styles.
- **Approach:**
  - Introduce a shared object map for supported burst class names and throw compile-time checks.
